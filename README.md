# Шахматка объектов — система учёта заселения

Веб-приложение (PWA) для отдела заселения строительной компании: гибкая структура объекта (корпуса → этажи → помещения), визуальная шахматка со статусами передачи, роли «администратор / пользователь», авторизация с одобрением доступа, данные синхронизируются в реальном времени через Firebase.

## Что внутри

- **Шахматка** по каждому корпусу: этажи по вертикали, помещения по горизонтали, цвет ячейки = статус. Для паркинга — отдельная сетка.
- **Конструктор структуры** (только админ): создание/удаление/переименование корпусов, этажей, помещений — без программиста.
- **Карточка помещения**: смена статуса в один клик, комментарий.
- **Статистика**: краткий блок над шахматкой (по текущему корпусу) + отдельная вкладка с полной сводной таблицей по всем корпусам и по типам помещений.
- **Фильтры**: по статусу, типу, поиск по номеру.
- **Роли**: администратор (полный доступ, структура, пользователи) и обычный пользователь (шахматка, статусы, комментарии).
- **Авторизация**: email + пароль, самостоятельная регистрация, доступ открывает администратор. Первый зарегистрированный пользователь автоматически становится администратором.
- **PWA**: устанавливается на телефон как приложение, работает оффлайн-кэш статики.

## Технологии

- React 18 + Vite
- Firebase Authentication (email/password)
- Firebase Firestore (хранение данных + обновления в реальном времени)
- Чистый CSS, без UI-фреймворков — мобайл-фёрст, крупные кнопки

## 1. Настройка Firebase (один раз)

1. Зайдите на [console.firebase.google.com](https://console.firebase.google.com) и создайте новый проект.
2. В разделе **Build → Authentication** включите способ входа **Email/Password**.
3. В разделе **Build → Firestore Database** создайте базу данных (можно в режиме production — правила безопасности уже прописаны в `firestore.rules`).
4. В разделе **Project settings → General → Your apps** нажмите «Add app» → веб-приложение (`</>`), задайте любое имя. Firebase покажет объект конфигурации вида:

```js
const firebaseConfig = {
  apiKey: "...",
  authDomain: "...",
  projectId: "...",
  storageBucket: "...",
  messagingSenderId: "...",
  appId: "..."
};
```

5. Откройте файл `src/firebase.js` в проекте и вставьте туда свои значения вместо `ВАШ_API_KEY` и т.д.

6. Установите Firebase CLI (если ещё не установлен) и опубликуйте правила безопасности Firestore:

```bash
npm install -g firebase-tools
firebase login
firebase init firestore   # выберите существующий проект, файл правил оставьте firestore.rules
firebase deploy --only firestore:rules
```

Без этого шага правила безопасности останутся стандартными (закрытыми), и приложение не сможет читать/писать данные.

## 2. Запуск локально (для разработки)

Требуется [Node.js](https://nodejs.org) версии 18 или новее.

```bash
cd zaselenie-app
npm install
npm run dev
```

Откройте адрес, который покажет терминал (обычно `http://localhost:5173`).

Первый зарегистрированный через форму пользователь автоматически получит права администратора и сразу доступ — без подтверждения (иначе некому было бы подтверждать первого). Все последующие регистрации будут ждать одобрения от администратора во вкладке **«Пользователи»**.

## 3. Публикация в интернет (чтобы открывать с любого телефона)

Самый простой вариант — Firebase Hosting (бесплатный тариф вполне достаточен для одного объекта):

```bash
npm run build
firebase deploy --only hosting
```

После деплоя Firebase выдаст ссылку вида `https://ваш-проект.web.app` — она и есть постоянный адрес приложения. Откройте её с телефона и через меню браузера выберите «Добавить на главный экран» — приложение установится как обычное мобильное приложение (PWA).

Альтернативы Firebase Hosting: Vercel или Netlify — туда можно просто загрузить папку `dist` после `npm run build`, либо подключить git-репозиторий для автодеплоя.

## 5. Как расширять дальше

- **Несколько объектов (ЖК) одновременно**: сейчас все данные привязаны к одному `PROJECT_ID = 'default-project'` в `Dashboard.jsx`. Чтобы добавить мульти-объектность, нужно сделать экран выбора объекта и хранить `projectId` в состоянии/URL вместо константы — структура Firestore (`projects/{projectId}/...`) уже это поддерживает.
- **Новые типы помещений**: список типов сейчас в `src/lib/constants.js` (`DEFAULT_UNIT_TYPES`). Можно вынести их в Firestore (`projects/{projectId}/settings`), чтобы админ добавлял свои типы через UI без правки кода.
- **Экспорт в Excel**: добавить кнопку на странице статистики, которая формирует .xlsx из данных через библиотеку типа SheetJS.
- **История изменений по помещению**: сейчас хранится только текущий статус и комментарий; для истории можно завести подколлекцию `units/{unitId}/history`.

## 6. Безопасность

- Пароли пользователей хранит и шифрует сам Firebase Authentication — приложение их не видит и не хранит.
- Доступ к данным в базе разграничен правилами `firestore.rules`: обычный пользователь физически не может изменить структуру или удалить помещение, даже если попытается обратиться к базе напрямую, минуя интерфейс.
- Прежде чем давать ссылку сотрудникам, обязательно выполните деплой правил (`firebase deploy --only firestore:rules`) — без этого шага защита не работает.
